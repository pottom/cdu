# ADR-001 ‚Äî Core/UI boundary, and how cdu tracks upstream gdu

**Status:** accepted
**Decision:** Strategy B ‚Äî source fork with an upstream tracking branch.
**Engine at time of writing:** gdu v5.36.1 (`github.com/dundee/gdu/v5`, Go 1.25).

---

## 1. Where the core ends and the UI begins

The good news first: **gdu's engine is genuinely UI-free.** No file under `pkg/`
imports tview or tcell. The scanner, the tree model, sorting, device enumeration
and deletion are all independent of the interface.

Engine, all public:

| Package | What it gives us |
|---|---|
| `pkg/analyze` | `Dir`/`File`, `ParallelAnalyzer`/`SequentialAnalyzer`/`StoredAnalyzer`, `top.go` (largest files) |
| `pkg/fs` | the `Item` interface, `SortBy`/`SortOrder` |
| `pkg/device` | mount points and their usage ‚Äî no internal deps at all |
| `pkg/remove` | `ItemFromDir`, `EmptyFileFromDir` |
| `report/` | `ReadAnalysis` (`-f`) and JSON export (`-o`) |
| `pkg/path`, `pkg/annex`, `pkg/timefilter` | path shortening, git-annex sizes, `--since`/`--max-age` |

Interfaces: `tui/` (tview), `stdout/` (non-interactive), `report/` (export).

**But the boundary is not a clean cut.** It runs *through* `internal/common`,
which Go forbids any external module from importing:

| Symbol | Location | Consequence for an out-of-tree UI |
|---|---|---|
| `Analyzer` interface | `internal/common/analyze.go` | cannot be named |
| `CurrentProgress`, `SignalGroup` | `internal/common/{analyze,signal}.go` | cannot be stored in a struct field |
| `ShouldDirBeIgnored`, `TimeFilter` | `internal/common/analyze.go` | reachable only via func literals |
| `CreateIgnoreFunc`, `CreateFileTypeFilter`, `CreateIgnorePattern` | `internal/common/ignore.go` | the whole ignore-pattern engine is unreachable |
| `common.UI` base struct | `internal/common/ui.go` | **embedded by `tui.UI`, `stdout.UI` and `report.UI`** |
| `app.UI` interface, with `SetAnalyzer(common.Analyzer)` | `cmd/gdu/app/app.go` | **an external UI can never satisfy it** |

The types that cross the boundary safely are `fs.Item`, `analyze.Dir`,
`analyze.File`, `device.Devices`, `fs.SortBy`/`SortOrder`.

## 2. Strategy A ‚Äî import gdu as a Go module

The brief preferred this, and it is the automation-friendly path: follow upstream
by bumping one line in `go.mod`. It does not survive contact with the code.

You *can* call the concrete analyzer ‚Äî `analyze.CreateAnalyzer()` returns an
exported `*ParallelAnalyzer`, and Go's internal rule restricts imports, not
values, so a func literal is assignable to `ShouldDirBeIgnored` and type
inference gives you the progress struct. What you cannot do is **name** any of
those types. That kills four things cdu actually needs:

1. **`--classic` is impossible.** Reusing `tui/` means constructing a `tui.UI`,
   which embeds `*common.UI`. Out of the question from another module.
2. **`app.UI` cannot be implemented.** Its method set includes
   `SetAnalyzer(common.Analyzer)`. `cmd/gdu/app` is therefore unusable as a
   library, and the entire flag/config/wiring layer would have to be rewritten.
3. **Byte-for-byte `stdout` and `-o` export parity is lost**, because both UIs
   embed `common.UI`. We would be reimplementing them and hoping the bytes match.
4. **The ignore-pattern engine must be rewritten** (~120 lines of regex assembly),
   and with it the risk of quietly diverging from gdu's `--ignore-dirs` semantics.

There is also no cancellation anywhere in the analyzer API ‚Äî no `context.Context`,
no `Stop()` ‚Äî which a wrapper cannot add from outside.

**And the API is not stable.** gdu has never bumped its major despite breaking its
own packages inside v5.3x:

- `fs.Item`: `GetItemCount() int` ‚Üí `int64`; `GetFiles()` ‚Üí `GetFiles(SortBy, SortOrder) iter.Seq[Item]`; `SetFiles` removed.
- `common.Analyzer`: `AnalyzeDir`'s third parameter changed meaning; `GetProgressChan()` was **removed** in v5.36 in favour of polling `GetProgress()`; `CurrentProgress.TotalSize` was renamed `TotalUsage`.

So Strategy A does not even buy the stability it promises. A `go.mod` bump would
still break the build on a minor release ‚Äî it would just break it with fewer
tools available to fix it.

## 3. Strategy B ‚Äî source fork (chosen)

We fork the tree. `internal/common` becomes ours, so every constraint above
evaporates: the Charm UI embeds `*common.UI` exactly like `tui.UI` does and
satisfies `app.UI` for free; `--classic`, `stdout` and `-o` export keep working
because **we do not touch them at all**.

The cost of a fork is merge pain, and we pay it down structurally.

### Branch model

- **`upstream`** ‚Äî verbatim `dundee/gdu`. Never touched. Must always fast-forward.
- **`upstream-cdu`** ‚Äî `upstream` plus exactly one commit, generated by
  `scripts/rename-upstream.sh`: the module path, `cmd/gdu` ‚Üí `cmd/cdu`, and the
  man page filename.
- **`main`** ‚Äî merges `upstream-cdu`, plus all cdu code.

**Putting the rename on its own branch is the load-bearing idea.** The rename
touches the import block of ~90 files. If it lived on `main`, every upstream
commit that adds or removes an import would conflict. Because the tracking branch
is *pre-renamed*, and the script is deterministic and idempotent, git only ever
sees upstream's real changes.

### Keeping the conflict surface small

The rule that makes this work, enforced by the root `AGENTS.md`:

- **Never edit** `pkg/`, `tui/`, `stdout/`, `report/`, `internal/common/`,
  `build/build.go`. These are replaced wholesale on every sync.
- **All new code goes in new directories** ‚Äî `charm/`, `internal/theme/`,
  `internal/trash/`, `internal/selfupdate/`, `build/cdu.go`. New files cannot
  conflict.
- **Only seven paths** are upstream files we deliberately modify, and they are
  where conflicts will concentrate: `go.mod`, `go.sum`, `cmd/cdu/main.go`,
  `cmd/cdu/app/app.go`, `Makefile`, `.github/workflows/*`, `README.md`, `cdu.1.md`.

A conflict outside that list means someone broke the rule.

### Verified

The rename was applied to v5.36.1 and the result **builds, vets clean, and passes
all 699 inherited tests**. Re-running the script is a no-op.

## 4. Automation

`.github/workflows/watch-upstream.yml` runs daily. On a new gdu release it
fast-forwards `upstream`, merges into `upstream-cdu`, re-runs the rename, merges
into a `chore/sync-gdu-vX.Y.Z` branch off `main`, updates `UPSTREAM_VERSION`, and
opens a PR.

Dependabot and Renovate are **not** applicable ‚Äî they bump module versions, which
is Strategy A's mechanism, not ours.

**Releases gate on green CI, and nothing auto-merges.** Given ¬ß2's evidence that
gdu breaks its own API on minor releases, an upstream change *will* eventually
break `charm/`. That failure mode is the normal one, not the exceptional one: the
PR stays open, CI is red, no tag is cut, and a human fixes it. "Fully automatic"
is only honest for compatible updates, and this pipeline says so out loud.

## 5. Consequences

- We carry gdu's whole tree, including code we never touch. Acceptable: it is the
  price of `--classic` and of byte-exact export parity, both of which the brief
  requires.
- We inherit gdu's dependencies (tview, tcell) even though the default UI does not
  use them. They stay linked in for `--classic`.
- Two gaps in the engine must be filled in cdu-owned code, because upstream has
  neither: **cancellation** (quitting mid-scan currently ends with the process)
  and **trash/undo** (`pkg/remove` only does a permanent `os.RemoveAll`).
- The engine version is recorded in `UPSTREAM_VERSION` and surfaced as SemVer
  build metadata: `cdu v0.1.0+gdu5.36.1`.
